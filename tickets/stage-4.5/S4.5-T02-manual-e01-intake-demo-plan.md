# S4.5-T02 - Case Workspace And First-Testing Command Plan

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Plan the first command-line orchestration workflow for user-provided E01 files, including case workspace creation/opening, evidence registration, concise command prompt output, and JSON artifacts.

This ticket is documentation-only. It prepares the implementation shape for a later coding ticket, but does not implement `app.backend.api.first_testing`.

## Command Target

The primary future command should be:

```powershell
python -m app.backend.api.first_testing X:\Evidence\CaseA\disk.E01 --case .test-artifacts\first-testing\case-a
```

The expanded form should be:

```powershell
python -m app.backend.api.first_testing --evidence-dir X:\Evidence\CaseA --first-segment disk.E01 --case .test-artifacts\first-testing\case-a --case-name "Case A first test" --actor "local-test"
```

Recommended arguments:

| Argument | Required | Purpose |
| --- | --- | --- |
| positional `.E01` path | one input form required | Direct selected first segment path |
| `--evidence-dir` | one input form required | Directory containing the segment set |
| `--first-segment` | required with `--evidence-dir` | Explicit `.E01` filename inside `--evidence-dir` |
| `--case` | required | Case workspace directory |
| `--output` | optional | Output/artifact directory; default should be `<case>\outputs` |
| `--case-name` | optional | Human-readable case name; default can derive from workspace folder |
| `--case-description` | optional | Optional local description |
| `--actor` | optional | Examiner/local actor string for audit events |
| `--adapter` | optional | `pyewf` by default; `stub` only for tests/development |
| `--json-only` | optional | Print machine-readable JSON instead of concise text |
| `--redact-paths` | optional | Redact evidence roots in command prompt summaries |

For Stage 4.5, selecting `.E02` or later as the primary path should be rejected with a structured invalid-input result that tells the user to select the `.E01` segment.

## Existing Code To Reuse

The future command should reuse the current code rather than inventing parallel models.

| Workflow piece | Existing surface | Intended use |
| --- | --- | --- |
| E01 intake | `app.backend.api.intake.run_e01_intake()` | Run the first evidence intake bundle and produce JSON-ready segment, adapter, metadata, verification, and warning fields |
| Segment discovery | `app.backend.forensic_core.discover_e01_segments()` | Continue to discover sibling `.E02/.E03...` filenames through the intake path |
| Case database | `app.backend.case_store.connect()` and `initialize_schema()` | Create/open `<case>\case.db` and ensure schema exists |
| Case row | `insert_case()` | Create the local case record |
| Evidence row | `insert_evidence_source()` | Persist the intake snapshot exactly as returned by `run_e01_intake()` |
| Audit row | `insert_audit_event()` | Record case creation/opening, evidence intake, artifact writes, unsupported parser sections, and command completion |
| Schema versions | `SCHEMA_VERSION` and `INTAKE_SCHEMA_VERSION` | Store tool/schema version fields in the run manifest |

No new parser, verifier, preview, export, hash, signature, or filesystem behavior should be hidden inside this orchestration command. Later tickets own those real-E01 bridges.

## Case Workspace Layout

Default workspace layout:

```text
.test-artifacts/
  first-testing/
    local-runs/
      case-a/
        case.db
        run-manifest.json
        command-summary.txt
        outputs/
          intake.json
          case.json
          audit.json
          unsupported-sections.json
          file-list.json
          file-list.csv
          reports/
          exports/
          manifests/
```

Rules:

- `--case` identifies the workspace root.
- `case.db` should live directly under `--case`.
- If `--output` is omitted, use `<case>\outputs`.
- The output folder must not be inside the evidence folder.
- The evidence folder must not be inside the output folder.
- The command must not write beside E01 segments.
- Local unredacted artifacts should stay under ignored paths such as `.test-artifacts/first-testing/`.
- Shared summaries may redact private paths, but local JSON should preserve the original paths for repeatability unless the user explicitly requests redacted local artifacts.

## Planned Command Flow

1. Resolve input:
   - accept either a direct `.E01` path or `--evidence-dir` plus `--first-segment`;
   - reject missing paths, unsupported extensions, directories passed as direct evidence, and `.E02+` primary selections;
   - compute the evidence root for overlap checks and optional redaction.

2. Prepare workspace:
   - create `--case` and output subfolders;
   - refuse evidence/output overlap before writing artifacts;
   - open `<case>\case.db` with `connect()`;
   - run `initialize_schema()`.

3. Create or open case:
   - first implementation may create a new case every run;
   - later work may add explicit `--case-id` or `--open-existing` behavior;
   - call `insert_case()` with `--case-name` and `--case-description`;
   - audit action: `first_testing.case_created`.

4. Run evidence intake:
   - call `run_e01_intake(selected_e01_path, adapter)`;
   - write `outputs\intake.json`;
   - call `insert_evidence_source()` with the intake result;
   - audit action: `first_testing.evidence_intake_completed`.

5. Record current unsupported sections:
   - verification may be `not_run`, `metadata_unavailable`, or `reader_not_implemented` depending on current adapter status;
   - real EWF metadata, real EWF verification, partition parsing, filesystem listing, real file metadata, real preview, real file export, real file hash/signature, and real file-list export must appear as unsupported/not-yet-implemented until later tickets change that truth;
   - write `outputs\unsupported-sections.json`.

6. Write case and audit artifacts:
   - write `outputs\case.json` with case id, evidence id, database path, schema version, and artifact paths;
   - write `outputs\audit.json` from `list_audit_events()`;
   - audit action: `first_testing.artifacts_written`.

7. Write run manifest and command summary:
   - write `<case>\run-manifest.json`;
   - write `<case>\command-summary.txt`;
   - print the same concise summary to the command prompt unless `--json-only` is used;
   - audit action: `first_testing.run_completed`.

## Run Manifest Fields

Recommended `run-manifest.json` fields:

| Field | Purpose |
| --- | --- |
| `schema_version` | Suggested value: `stage4_5.first_testing_run.v1` |
| `run_id` | Stable generated id for this run |
| `started_at` / `ended_at` | UTC timestamps |
| `status` | Overall status such as `ok_with_unsupported_sections`, `invalid_input`, `intake_failed`, or `error` |
| `command` | Arguments used, with optional redaction for shared summaries |
| `working_directory` | Local working directory, optional/redacted in shared summary |
| `tool_versions` | Python version, intake schema version, case-store schema version, app/stage label |
| `input` | Selected path, input form, evidence root, first segment, redaction policy |
| `case` | Case workspace, database path, case id, case name |
| `evidence` | Evidence id, intake status, read-only assertion, segment count |
| `artifacts` | Paths to `intake.json`, `case.json`, `audit.json`, `unsupported-sections.json`, `command-summary.txt`, file-list/export/report placeholders |
| `sections` | Status for segment discovery, metadata, verification, volume discovery, filesystem navigation, file metadata, preview, analysis, export, file-list export |
| `warnings` | Combined structured warnings from intake and orchestration checks |
| `unsupported` | Explicit list of features not implemented in the current code path |

## Command Prompt Summary

Default text output should be short enough to read in PowerShell. Suggested shape:

```text
Stage 4.5 first test
Case: Case A first test
Workspace: .test-artifacts\first-testing\case-a
Evidence: <EVIDENCE_ROOT>\disk.E01

Intake: reader_not_implemented
Segments: 3 discovered, 0 missing warnings
Adapter: pyewf available=false dependency=unavailable
Metadata: unavailable
Verification: not run

Current unsupported sections:
- real EWF metadata
- real EWF verification
- partition/filesystem parsing
- E01-backed file preview/export/hash/signature

Artifacts:
- run-manifest.json
- outputs\intake.json
- outputs\case.json
- command-summary.txt
```

If `--json-only` is passed, print the manifest JSON or a compact run result JSON. Local files should still be written unless a later ticket adds a dry-run mode.

## Current Behavior Versus Desired Stage 4.5 Behavior

The first implementation of this command can be useful before real parsers exist, but it must label its limits.

| Area | Current-code-backed behavior available now | Later Stage 4.5 target |
| --- | --- | --- |
| Case workspace | Can create SQLite DB and insert case/evidence/audit rows through existing helpers | Add open-existing behavior and richer run history if needed |
| Segment info | Can discover sibling segment filenames and missing/gap warnings | Keep as first visible proof against real user paths |
| EWF metadata | Adapter boundary exists, but real metadata is not implemented | S4.5-T03 investigates/implements real metadata if feasible |
| Verification | Result shape exists, but real verification is not implemented | S4.5-T03 defines real verification path or clear unsupported result |
| File navigation | Stub/listing shapes exist, but not E01-backed | S4.5-T04 adds EWF stream, partition, and filesystem bridge |
| Preview/export/hash/signature | Provider-backed surfaces exist, but not E01-backed | S4.5-T05 connects selected filesystem files to preview/export/analysis providers |
| File-list export | JSON-friendly listing shapes exist | S4.5-T06 defines JSON/CSV export from real parser results |
| Visual output | Not implemented | S4.5-T06 may add static HTML or a readable summary bundle |

## Future Implementation Tickets

This plan implies a later implementation ticket for the command shell itself. That future ticket should be narrow:

- add `app/backend/api/first_testing.py`;
- add tests with tiny dummy `.E01/.E02` filenames only, not real evidence;
- create case/output folders under `.test-artifacts/`;
- persist intake results to SQLite;
- write the manifest and summary artifacts;
- print unsupported sections honestly;
- update docs and manual-test instructions.

It should not implement real `pyewf`, real verification, partition parsing, filesystem parsing, E01-backed file content, preview/export/hash/signature, file-list export, static HTML, search/timeline, UI, packaging, deleted recovery, or carving.

## Acceptance Criteria

- The plan gives the user one concrete command target.
- The plan defines accepted arguments and workspace layout.
- The plan distinguishes current behavior from desired Stage 4.5 behavior.
- The plan avoids claiming real metadata/verification until implemented.
- The plan identifies exactly which current functions are reused for case creation and intake.
- The plan gives a future coding agent enough detail to implement only the command shell without broad parser work.
