## Objective

Fix the S4.5-IMP02 review finding where `stored_hash_not_verified` can cause the adapter to also emit `metadata_partial` even when no metadata field was missing or failed.

This is a small correction ticket for warning honesty only. Keep the S4.5-IMP02 implementation intact, preserve public result shapes, and do not expand Stage 4.5 scope.

## Review Finding

In `app/backend/forensic_core/ewf_reader.py`, `_extract_metadata()` currently appends `metadata_partial` whenever the local metadata warning list is non-empty. Because stored EWF hash values add `stored_hash_not_verified`, a fully successful metadata read that includes stored hashes can be mislabeled as partial.

That matters because Stage 4.5 first-testing output will be used as human review evidence. `metadata_partial` should mean metadata was incomplete or a metadata field failed/unavailable, not merely that stored hash metadata was handled conservatively.

## Implementation Notes - 2026-07-17

Result: reviewed and done.

Implemented:

- Changed `_extract_metadata()` so `metadata_partial` is emitted only when a `metadata_field_unavailable` warning is present.
- Preserved `stored_hash_not_verified` as a separate metadata warning when stored EWF hash values are read.
- Preserved verification warnings such as `verification_not_supported`, `verification_failed`, and `verification_error` as separate verification signals, not metadata-partial triggers.
- Added dependency-free fake-`pyewf` regression assertions proving complete metadata plus stored hashes does not emit `metadata_partial`, while true partial metadata still does.

Verification:

## Required Work

- Change metadata warning construction so `metadata_partial` is emitted only when actual metadata fields are missing, unavailable, unsupported, or fail to read.
- Do not emit `metadata_partial` solely because:
  - stored hash metadata was read and `stored_hash_not_verified` was emitted;
  - verification is unsupported, failed, or not run;
  - dependency version is unknown, unless the implementation intentionally treats that as partial metadata and documents/tests that choice.
- Preserve `stored_hash_not_verified` exactly as a separate warning when stored EWF hashes are read.
- Preserve `verification_not_supported`, `verification_failed`, and `verification_error` as verification warnings, not metadata-partial signals.
- Keep missing `pyewf` behavior unchanged.
- Keep first-testing artifacts unchanged except for corrected warnings.
- Do not implement streams, partition/filesystem parsing, E01-backed selected-file providers, file-list output, static HTML, search/timeline, UI, reports, deleted recovery, carving, packaging, or native dependency installation.
