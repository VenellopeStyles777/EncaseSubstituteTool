# S4.5-IMP09 - Nested Directory Navigation Demo

Status: Done

Stage: Stage 4.5 - first testing implementation runway extension

Owner: VS Code coding agent

Reviewer: Research/review agent

## Objective

Add explicit nested directory navigation for the real E01-backed filesystem demo.

The user tested the demo and found that seeing partitions/root status is not enough. Before Stage 5, the command-line demo must prove that the project can navigate into at least one actual directory inside the image and show entries from the filesystem, not just partition records or root-level counts.

This ticket should add narrow, explicit directory navigation. It is not a broad crawl or search index.

This is a hard Stage 4.5 demo requirement. Do not downgrade it into documentation wording or a root-listing restatement. The user needs to be able to run a command, open the generated artifacts, and see entries from inside an actual directory in the E01-backed filesystem.

## Required Behavior

Add explicit directory navigation to first testing.

Required CLI options:

- `--list-directory-path "<path-from-root>"`;
- `--demo-list-first-directory`.

Optional, only if the current `file_id` shape can be resolved safely:

- `--list-directory-id "<parser-backed-directory-id>"`.

Suggested real-image smoke command:

```powershell
.\.python312-embed\python.exe -m app.backend.api.first_testing --evidence-dir ".\ Test Image" --first-segment "C16242-1-RL1-E003.E01" --case ".test-artifacts\first-testing\nested-navigation-real-image" --output ".test-artifacts\first-testing\nested-navigation-real-image\outputs" --demo-list-first-directory --redact-paths
```

Rules:

- Accept at most one directory selector mode. `--list-directory-path`, `--list-directory-id`, and `--demo-list-first-directory` must be mutually exclusive if all are present.
- Only list direct children of the requested directory by default.
- `--demo-list-first-directory` may probe readable allocated directories already visible from the root listing until it finds one with entries. This must stay bounded to root-directory candidates and must not recurse through the whole volume.
- Do not recursively crawl unless a later reviewed ticket asks for it.
- Do not read file contents.
- Do not claim navigation is real-parser-backed unless entries come from the EWF stream plus parser APIs.
- Shared handoffs must report counts/statuses/type counts only, not real filenames or sensitive internal paths.
- If a requested path exists but is a file, return a structured `path_not_directory` status.
- If a requested path does not exist or cannot be opened, return structured status/warnings rather than fabricating entries.

Implementation notes to consider:

- `app/backend/api/directory_listing.py` currently treats nested paths as `path_unsupported`; this ticket should replace that limitation for real parser-backed pytsk3 listings.
- `Pytsk3FilesystemAdapter.inspect_volume()` currently parses root entries with `FS_Info.open_dir(path="/")`; this ticket likely needs a parser-backed directory-listing method or helper that can call `open_dir(path=<requested>)`.
- The existing stub adapter may remain root-only or gain deterministic nested test data, but stub output must never be labeled real-parser-backed.
- Keep the existing root-listing artifacts intact; nested directory listing should be an additional artifact and summary field.
