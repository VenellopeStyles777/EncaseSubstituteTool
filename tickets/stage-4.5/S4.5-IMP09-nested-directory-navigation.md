# S4.5-IMP09 - Nested Directory Navigation Demo

Status: Draft

Stage: Stage 4.5 - first testing implementation runway extension

Owner: VS Code coding agent

Reviewer: Research/review agent

## Objective

Add explicit nested directory navigation for the real E01-backed filesystem demo.

The user tested the demo and found that seeing partitions/root status is not enough. Before Stage 5, the command-line demo must prove that the project can navigate into at least one actual directory inside the image and show entries from the filesystem, not just partition records or root-level counts.

This ticket should add narrow, explicit directory navigation. It is not a broad crawl or search index.

## Scope Boundary

Allowed:

- extend the parser-backed filesystem adapter to list a requested directory below root;
- add first-testing CLI options for an explicit directory path/id or a safe demo directory mode;
- write nested directory-listing artifacts;
- update command summary, manifest, file-list/HTML inventory, and manual guide;
- run a real-image smoke that proves at least one nested directory can be listed.

Not allowed:

- no recursive full-volume crawl;
- no search/timeline index;
- no file-content extraction unless already covered by explicit selected-file behavior;
- no selected-file auto-export/preview/hash/signature;
- no deleted recovery/carving;
- no UI/report system/PDF;
- no real evidence commits.

## Required Behavior

Add explicit directory navigation to first testing.

Suggested CLI options:

- `--list-directory-path "<path-from-root>"`;
- `--list-directory-id "<parser-backed-directory-id>"`;
- optional `--demo-list-first-directory` for a local smoke that chooses a readable allocated root directory and lists it without exporting content.

Rules:

- Accept either path or id, not both.
- Only list direct children of the requested directory by default.
- Do not recursively crawl unless a later reviewed ticket asks for it.
- Do not read file contents.
- Do not claim navigation is real-parser-backed unless entries come from the EWF stream plus parser APIs.
- Shared handoffs must report counts/statuses only, not real filenames or sensitive internal paths.

## Artifacts

Write:

- `outputs/directory-listing.json`
- optional `outputs/directory-listing.csv`
- optional `outputs/navigation-readiness.json`

Minimum `directory-listing.json` fields:

- `status`;
- `source_kind`;
- `parser_backing`;
- `volume_id`;
- `filesystem_id` if available;
- `requested_path` or `requested_file_id`;
- `resolved_path` or `resolved_file_id`;
- `entry_count`;
- `entries`;
- `warnings`;
- `provenance`;
- `read_only_asserted`;
- `source_modified`.

Each entry should preserve:

- file id/path/name;
- entry type;
- size where available;
- allocation/deleted flags where available;
- timestamps where available;
- parser/source status and warnings.

## Real-Image Requirement

Run a real-image nested-directory smoke against the ignored local ` Test Image/` set.

The smoke must show, without quoting names:

- command exits 0;
- root filesystem remains real-parser-backed;
- one actual directory inside the image is selected explicitly or by the safe demo mode;
- nested directory listing status is `ok`;
- nested entry count is greater than zero;
- at least one nested entry is a regular file or directory if available;
- selected-file preview/export/hash/signature remain opt-in and not run unless explicitly requested;
- source modified is false and read-only is asserted.

If the local image has no readable nested directories, mark the ticket `Blocked` with evidence, do not fabricate a result, and keep Stage 5 blocked.
