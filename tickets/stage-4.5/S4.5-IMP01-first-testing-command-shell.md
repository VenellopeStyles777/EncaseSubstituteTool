## Objective

Implement the first safe command-line workflow for user-provided E01 first testing.

This slice should create a narrow orchestration command that accepts a selected `.E01`, creates a case workspace, persists the existing Stage 1 intake result into the case store, writes a run manifest and concise summary, and records the currently unsupported real-E01 sections honestly.

It must not implement real EWF metadata, real EWF verification, EWF-backed streams, partition/filesystem parsing, E01-backed file content providers, file-list export, static HTML, search, timeline, UI, reports, deleted recovery, carving, or packaging.

## Required Behavior

Add `python -m app.backend.api.first_testing`.

Accepted input forms:

- Direct first segment:

  ```powershell
  python -m app.backend.api.first_testing X:\Evidence\CaseA\disk.E01 --case .test-artifacts\first-testing\case-a
  ```

- Evidence directory plus explicit first segment:

  ```powershell
  python -m app.backend.api.first_testing --evidence-dir X:\Evidence\CaseA --first-segment disk.E01 --case .test-artifacts\first-testing\case-a
  ```

Supported arguments:

- positional `.E01` path, mutually exclusive with `--evidence-dir`;
- `--evidence-dir`;
- `--first-segment`, required with `--evidence-dir`;
- `--case`, required workspace directory;
- `--output`, optional artifact directory, default `<case>/outputs`;
- `--case-name`, optional, default derived from the case workspace folder;
- `--case-description`, optional;
- `--actor`, optional audit actor;
- `--adapter`, choices `pyewf` and `stub`, default `pyewf`;
- `--json-only`, optional, prints JSON instead of human summary;
- `--redact-paths`, optional, redacts evidence root in console/summary text while preserving local JSON paths.

Input and path safety:

- Accept only a selected `.E01` as the primary evidence input for this first implementation.
- Reject `.E02` or later as the primary input with a structured `invalid_input` result telling the user to select `.E01`.
- Reject unsupported extensions, missing direct paths, missing `--first-segment`, direct directories, and conflicting input forms.
- Refuse evidence/output overlap before writing artifacts:
  - output directory must not be inside the evidence directory;
  - evidence directory must not be inside the output directory;
  - case workspace must not be inside the evidence directory;
  - evidence directory must not be inside the case workspace.
- Never write beside evidence segments.
- Treat evidence as read-only and report `source_modified: false` / `read_only_asserted: true` in run output when only current read-only helpers were used.

Workspace and persistence:

- Create the case workspace at `--case` only after input and overlap checks pass.
- Create/open `<case>/case.db` with `connect()` and `initialize_schema()`.
- Create a case row with `insert_case()`.
- Run `run_e01_intake(selected_e01_path, adapter)`.
- Persist the intake result with `insert_evidence_source()`.
- Insert audit events for:
  - `first_testing.case_created`;
  - `first_testing.evidence_intake_completed`;
  - `first_testing.artifacts_written`;
  - `first_testing.run_completed`.
- Use `list_audit_events()` for the audit artifact.

Required artifacts:

- `<case>/run-manifest.json`
- `<case>/command-summary.txt`
- `<output>/intake.json`
- `<output>/case.json`
- `<output>/audit.json`
- `<output>/unsupported-sections.json`

Do not create `file-list.json`, `file-list.csv`, static HTML, export artifacts, report artifacts, or fake parser outputs in S4.5-IMP01. They belong to later slices.

Run manifest:

- Use schema version `stage4_5.first_testing_run.v1`.
- Include run id, timestamps, command/input summary, tool/schema versions, case workspace/database paths, case id, evidence id, artifact paths, intake status, segment count, adapter/dependency status, warnings, unsupported sections, read-only assertion, `source_modified: false`, and overall status.
- Suggested overall statuses:
  - `ok_with_unsupported_sections` when orchestration and intake persistence succeed, even if real metadata/verification are unavailable;
  - `invalid_input` for input validation failures;
  - `unsafe_output_path` for overlap/safety failures;
  - `intake_failed` for unexpected intake/persistence failures if handled.

Unsupported sections:

Write `unsupported-sections.json` with explicit rows for at least:

- real EWF metadata: owner `S4.5-IMP02`;
- real EWF verification: owner `S4.5-IMP02`;
- EWF-backed byte stream: owner `S4.5-IMP03`;
- partition/filesystem parsing: owner `S4.5-IMP03`;
- E01-backed preview/export/hash/signature content: owner `S4.5-IMP04`;
- file-list JSON/CSV output: owner `S4.5-IMP05`;
- static HTML summary: owner `S4.5-IMP05`;
- implementation handoff/manual-test reconciliation: owner `S4.5-IMP06`.

Each row should include a stable `section`, `status: "not_implemented"`, `owner`, and a concise message. These are honesty markers, not fake outputs.

Command output:

- Default output should be a concise readable summary with case/workspace, evidence path or redacted evidence path, intake status, segment count, adapter/dependency status, unsupported sections, and artifact paths.
- `--json-only` should print JSON for the run result/manifest.
- `--redact-paths` should redact the evidence root in command summary text and console output as `<EVIDENCE_ROOT>`, while local JSON artifacts preserve original paths.

## Acceptance Criteria

- A user can run a first-testing command against dummy or real local `.E01` paths and get a case workspace, persisted intake snapshot, manifest, summary, audit artifact, and unsupported-section artifact.
- The command reuses existing intake and case-store helpers.
- Evidence/output overlap is refused before writing artifacts.
- The command never writes beside evidence and does not modify evidence.
- Output honestly labels real metadata, verification, filesystem parsing, file content, file-list output, and static HTML as not implemented.
- Automated tests cover the command without real evidence or native forensic dependencies.
- No Stage 5 search/timeline, parser bridge, UI, report, deleted recovery, carving, packaging, or native dependency requirement is added.
