Status: Draft

Stage: Stage 4.5 - first testing implementation runway

Owner: VS Code coding agent

Reviewer: Research/review agent

## Objective

Implement E01-backed selected-file content providers for preview, export, hash, and signature analysis by reusing existing Stage 2 through Stage 4 surfaces.

This ticket starts only after S4.5-IMP03 can produce real parser-backed filesystem entries or explicit metadata-only/unavailable statuses.

Do not implement file-list JSON/CSV, static HTML, search/timeline, UI, reports, deleted recovery, carving, or packaging.

- Existing `preview_file()`

## Required Behavior

Add a shared selected-file content reader plus thin provider wrappers, or a local equivalent that preserves the same separation:

- preview provider wrapper;
- export provider wrapper;
- analysis provider wrapper.

Rules:

- Do not fall back to stub providers when claiming E01-backed content.
- Use `source_kind="real_parser"` only when bytes are proven to come from parser-backed E01 filesystem entries.
- Preserve case/evidence/volume/filesystem/file provenance.
- Implement an explicit size/streaming policy before export/hash of large real files.
- Represent deleted/sparse/special/encrypted/compressed/unreadable entries as structured statuses.
- Do not silently load large evidence files into memory.

## First-Testing Command Integration

Allowed artifact additions:

- `<output>/selected-file-preview.json`
- `<output>/selected-file-analysis.json`
- `<output>/selected-file-export.json`
- export manifest artifacts only for explicitly selected safe files

The first implementation may require an explicit selected-file argument or use a small safe fixture/fake parser entry in tests. It must not automatically export/hash arbitrary files from real evidence.

Do not create full file-list JSON/CSV or static HTML in S4.5-IMP04.

## Tests

Future implementation should cover:

- preview wrapper returns bounded real-parser-labeled content;
- export wrapper reuses `export_file()` and verifies written output;
- analysis wrapper reuses existing hash/signature functions;
- no E01-backed claim uses stub bytes;
- metadata-only entries report content unavailable;
- directories report `path_not_file`;
- deleted entries report deleted recovery deferred unless recoverable bytes are proven;
- large files are refused or streamed per policy;
- provider exceptions become structured statuses/warnings;
- provenance survives all wrapper conversions.
