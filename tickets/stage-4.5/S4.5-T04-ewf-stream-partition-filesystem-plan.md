# S4.5-T04 - EWF Stream, Partition, And Filesystem Plan

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Plan the missing bridge from user-provided E01 files to navigable volumes and real filesystem entries.

This ticket is documentation-only. It does not implement an EWF stream, partition parser, or real filesystem parser.

## Planned Data Path

The intended real-E01 path should be:

```text
selected .E01 path
  -> segment discovery
  -> EWF metadata/dependency status
  -> EwfImageByteStream implementing ImageByteStream
  -> discover_volumes(..., strategy="partition_table" or "auto")
  -> VolumeInfo records with source path, stream type, offsets, lengths
  -> Pytsk3FilesystemAdapter over the selected volume
  -> FilesystemResult and FilesystemEntry records
  -> list_directory() response shape
```

Important boundary:

- EWF metadata/verification belongs to `PyewfEwfReaderAdapter`.
- EWF byte access should be a sibling stream implementation, not hidden inside metadata reading.
- Partition parsing should produce `VolumeInfo` records only.
- Filesystem parsing should produce `FilesystemResult` / `FilesystemEntry` records only.
- File byte extraction for preview/export/hash/signature belongs to S4.5-T05, not this ticket.

## EWF-Backed Image Stream Plan

Add a future `EwfImageByteStream` or equivalent in a reviewed implementation ticket.

Expected behavior:

- implement the existing `ImageByteStream` protocol;
- use `stream_type="ewf"`;
- expose `describe()` as `ImageStreamInfo`;
- expose `read_at(offset, length)` as `ImageReadResult`;
- open user-provided segment sets read-only;
- preserve the selected `.E01` path as `source_path`;
- keep all segment paths in stream-private state or a manifest, not in a new result shape unless needed;
- use EWF media size as `ImageStreamInfo.size`;
- reject negative offsets/lengths before reader calls;
- return structured `read_beyond_end` and `read_truncated_at_eof` behavior consistent with `LocalFileImageStream`;
- return structured dependency/open/read errors without raw tracebacks;
- close reader handles deterministically.

Recommended stream status/warning codes:

| Code | Meaning |
| --- | --- |
| `dependency_unavailable` | `pyewf`/libewf cannot be imported or opened |
| `ewf_open_failed` | Segment set could not be opened |
| `ewf_media_size_unavailable` | Reader opened but logical size could not be determined |
| `ewf_read_failed` | Reader failed during `read_at` |
| `read_beyond_end` | Offset is beyond logical media size |
| `read_truncated_at_eof` | Requested range extends past logical media end |
| `segment_set_incomplete` | Segment discovery found a gap or missing segment |

Do not reuse `LocalFileImageStream` for E01 bytes. An `.E01` file is a container; reading the first segment as a local raw file would be misleading.

## Partition Discovery Plan

Evolve `discover_volumes()` by adding a reviewed partition strategy while preserving current whole-image behavior.

Recommended strategy values:

| Strategy | Behavior |
| --- | --- |
| `whole_image` | Current behavior; one volume from offset `0` to stream size |
| `partition_table` | Attempt real partition table parsing |
| `auto` | Attempt partition parsing first, then return a structured no-table or whole-image fallback depending on policy |

Recommended statuses:

| Status | Meaning |
| --- | --- |
| `ok` | One or more volumes discovered |
| `image_stream_unavailable` | Stream could not be described/read |
| `partition_dependency_unavailable` | `pytsk3`/TSK partition support is unavailable |
| `partition_parsing_unsupported` | Strategy is known but not implemented yet |
| `partition_table_not_found` | Parser ran but found no partition table |
| `partition_parse_error` | Parser attempted and failed |
| `partition_parse_partial` | Some volumes were emitted, but parser also reported warnings/errors |
| `empty_image` | Stream size is zero |

`VolumeInfo` should remain the stable output shape. Future real partitions should fill:

- `volume_id`: deterministic id such as `volume-<index>` or a stable source/evidence-derived id;
- `volume_index`: parser order;
- `offset`: byte offset from the start of the logical image;
- `length`: byte length when known;
- `volume_type`: partition type or table label when available;
- `description`: human-readable partition description;
- `read_only`: combined stream/parser read-only assertion;
- `warnings`: partition-specific uncertainty.

Offsets and lengths must be byte values in project result shapes. If a parser reports sectors, convert using an explicitly known sector size and record warnings when sector size is unknown.

## Pytsk3 Filesystem Adapter Plan

`Pytsk3FilesystemAdapter` eventually needs to consume the selected `VolumeInfo` plus an image reader for the corresponding source stream.

Recommended design:

- keep `FilesystemResult`, `FilesystemEntry`, and `list_directory()` response shape stable;
- preserve `inspect_volume(volume)` as the public adapter method if possible;
- inject an image-stream or image-info factory into the adapter constructor for tests and real EWF/raw support;
- build a pytsk-compatible image object around `ImageByteStream.read_at()` when parsing EWF-backed volumes;
- open filesystem parsing at `volume.offset`;
- copy `volume.offset` and `volume.length` into every `FilesystemEntry`;
- expose only metadata in T04; do not expose file bytes yet;
- make nested directory traversal a later scoped extension if root listing is the first implementation slice.

Planned metadata mapping for entries:

| `FilesystemEntry` field | Planned source |
| --- | --- |
| `file_id` | Stable parser inode/meta address plus volume id |
| `path` | Normalized absolute path beginning with `/` |
| `name` | Display name from directory entry |
| `entry_type` | `file`, `directory`, `symlink`, `special`, or `unknown` |
| `size` | Logical file size when available |
| `allocated` | Parser allocation flag when available |
| `deleted` | Parser deleted/unallocated flag when available |
| `timestamps` | Created, modified, accessed, metadata-changed values when available |
| `filesystem_type` | Parser filesystem type label |
| `adapter_name` | `pytsk3-filesystem-adapter` |
| `read_only` | Combined stream/adapter assertion |

Timestamp conversion should preserve uncertainty. If timezone handling is unclear, store normalized UTC only when justified and include original/parser details in warnings or future metadata.

## Dependency And Parser Capability Behavior

The first-testing command should show separate capability rows:

| Capability | Dependency | Current expected state |
| --- | --- | --- |
| EWF byte stream | `pyewf`/libewf | unavailable locally |
| Partition parsing | `pytsk3`/The Sleuth Kit or equivalent | unavailable locally |
| Filesystem parsing | `pytsk3`/The Sleuth Kit | unavailable locally |

Do not collapse these into a single generic parser failure. A user should be able to tell whether EWF opening, partition parsing, or filesystem parsing blocked the workflow.

Security note: real filesystem parsers process untrusted evidence. Future real-parser tickets should keep parser actions read-only, bounded where practical, and review whether process isolation/sandboxing is needed for manual runs.

## Automated Test Strategy

Default tests should remain dependency-free and real-evidence-free.

Future implementation coverage should use fakes/mocks for:

- `EwfImageByteStream.describe()` success, dependency unavailable, open failure, missing media size, and read failure;
- `EwfImageByteStream.read_at()` valid reads, negative ranges, zero-length reads, EOF truncation, and read beyond end;
- `discover_volumes(strategy="partition_table")` dependency unavailable, no partition table, parse error, partial parse, and successful fake partition results;
- deterministic `VolumeInfo` ids, byte offsets, lengths, read-only assertions, and warnings;
- `Pytsk3FilesystemAdapter` dependency unavailable and importable-but-not-implemented behavior staying structured;
- future fake filesystem root entries mapping into `FilesystemEntry`;
- `list_directory()` preserving existing output shape when real adapter entries are supplied;
- path normalization and non-root listing behavior.

Optional integration tests with real E01 files must be skipped unless the user provides local paths and explicitly opts in. They must never be part of default `python -m pytest`.
