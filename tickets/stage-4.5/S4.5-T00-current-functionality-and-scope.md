## Minimum First-Testing Target

The desired Stage 4.5 direction is a command-line workflow similar to:

```powershell
python -m app.backend.api.first_testing path\to\evidence.E01 --case .test-artifacts\first-testing\case-001 --output .test-artifacts\first-testing\case-001\outputs
```

At the bare minimum, that future command should be able to:

- create or open a case workspace and store basic case/evidence information;
- discover E01 segment chain information;
- read real EWF metadata when the dependency is available, otherwise show dependency status;
- run or honestly report verification status;
- list discovered volumes and parser status;
- navigate file structure when filesystem parsing is available;
- show file metadata for selected entries;
- preview selected file bytes in hex/raw/text where supported;
- hash selected file bytes and detect signatures;
- export selected files with hashes and provenance manifests;
- export a file list as JSON and preferably CSV.

Current code does not yet meet that target end-to-end. It has useful pieces, but several pieces are still stubbed or provider-backed rather than real E01-backed.

## Additional Basic EnCase-Like Items To Add To The Target

For a very basic EnCase-style command-line MVP, add these to the list:

- audit log entries for case creation, evidence intake, verification, preview, analysis, and export actions;
- clear parser/dependency status summary so unsupported features are visible;
- output manifest bundle for each run, including command inputs, timestamps, tool version/schema versions, warnings, and hashes;
- file-list export to JSON plus CSV for quick examiner review;
- selected-item report or simple HTML summary after a run;
- timestamp normalization policy, including timezone handling and unknown/missing timestamp states;
- safe output-directory checks so exports never write beside evidence;
- redaction guidance for private evidence paths in shared logs/screenshots.
