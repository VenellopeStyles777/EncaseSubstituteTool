# S4.5-T01 - User-Provided E01 Handling Plan

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Define how user-provided E01 evidence files will be used for manual testing without committing evidence, leaking sensitive paths unnecessarily, or modifying source files.

## Required Work

- Define accepted input forms: direct `.E01` path, folder containing segment set, or ignored local config.
- Define ignored output locations for manual test artifacts such as `.test-artifacts/first-testing/`.
- Define evidence safety checks:
  - read-only access;
  - no writes beside evidence;
  - no output directory overlapping evidence path;
  - clear warnings for missing sibling segments.
- Define documentation rules for screenshots/transcripts so sensitive paths can be redacted.
- Define how manual test results should be logged in `review.md` or a dedicated manual-test log without recording private evidence content.

## Accepted Input Forms

Stage 4.5 should support these input forms in order of priority:

1. Direct selected first-segment path:

   ```powershell
   python -m app.backend.api.first_testing X:\Evidence\CaseA\disk.E01 --case .test-artifacts\first-testing\case-a
   ```

   The command should treat the selected path as the first segment and use existing segment discovery to locate sibling `.E02/.E03...` files in the same directory.

2. Folder plus explicit first segment name:

   ```powershell
   python -m app.backend.api.first_testing --evidence-dir X:\Evidence\CaseA --first-segment disk.E01 --case .test-artifacts\first-testing\case-a
   ```

   This form is useful when paths are long or when a later wrapper wants to keep directory and filename fields separate.

3. Ignored local run configuration:

   ```text
   .test-artifacts/first-testing/local-runs/case-a/run.json
   ```

   A local config can help repeat manual tests, but it must stay ignored and must not be required for automated tests. It may contain private local paths and should not be committed.

Do not support selecting `.E02` or later segments as the primary input in the first-testing command. If a later segment is supplied, the command should return a structured invalid-input result with guidance to select `.E01`.

## Local Evidence And Output Locations

Recommended local layout:

```text
.test-artifacts/
  first-testing/
    local-runs/
      case-a/
        case.db
        run-manifest.json
        intake.json
        command-summary.txt
        file-list.json
        file-list.csv
        exports/
        reports/
```

User-provided E01 files should normally live outside the repository. If the user chooses to place copies inside the workspace for convenience, they should be under `.test-artifacts/first-testing/evidence-local/` so Git ignores them.

The command should reject or warn on output directories that overlap the evidence directory. Exported files, manifests, file lists, logs, summaries, and reports should all be written under the explicit case/output directory.

## Evidence Safety Requirements

The first-testing workflow must:

- open E01 segments read-only;
- never write beside evidence segments;
- never rename, delete, normalize, repair, or resequence evidence segments;
- discover sibling segments by directory listing only until the EWF adapter reads metadata;
- report missing sibling segments as warnings or non-ok status, not raw tracebacks;
- report unsupported selected extensions, including `.E02` as the primary path, as structured invalid input;
- refuse output paths nested inside the evidence folder;
- refuse evidence folders nested inside the output folder;
- preserve original source paths in local JSON, while allowing redaction in shared summaries.

## Privacy And Redaction Rules

Manual-test artifacts may contain sensitive paths and metadata. Use these rules:

- Do not paste full private evidence paths into shared review notes unless the user explicitly approves.
- Redact source roots as `<EVIDENCE_ROOT>` in summaries and screenshots.
- Redact user names, client names, case names, serial numbers, examiner names, and acquisition notes unless the user explicitly approves.
- Keep local unredacted JSON under `.test-artifacts/first-testing/`.
- In `review.md`, record only sanitized command examples and high-level result counts unless the user asks for full details.
- If a screenshot or HTML summary is shared, confirm it does not show private paths or metadata first.

## Manual-Test Logging Plan

Stage 4.5 implementation handoffs should include a short manual-test section:

```text
Manual E01 test:
- Real E01 used: yes/no
- Evidence path redacted as: <EVIDENCE_ROOT>\disk.E01
- Command run:
- Output directory:
- Source modified: no
- Artifacts produced:
- Skipped pieces:
- Notes:
```

`functionality.md` manual-test status should remain `Untested` until the user confirms a real manual run. Automated tests and mocked dependency tests do not count as manual testing.

## Acceptance Criteria

- A reviewer can tell where user-provided E01 files should live and where outputs should be written.
- The plan prevents accidental Git tracking of evidence or sensitive output.
- The plan states which file paths may appear in command output and how to redact them for shared docs.
- The plan defines accepted input forms and invalid input behavior.
- The plan defines manual-test logging expectations without requiring real E01 files during automated tests.
