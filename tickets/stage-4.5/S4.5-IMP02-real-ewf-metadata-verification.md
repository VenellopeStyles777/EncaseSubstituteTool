# S4.5-IMP02 - Real EWF Metadata And Verification

Status: Ready

Stage: Stage 4.5 - first testing implementation runway

Owner: VS Code coding agent

Reviewer: Research/review agent

## Objective

Implement the smallest safe real-`pyewf` metadata and verification slice.

This ticket should upgrade the existing `PyewfEwfReaderAdapter` from dependency-safe skeleton to a best-effort real metadata reader when `pyewf` is importable, while preserving all missing-dependency behavior and dependency-free tests. It should also expose verification status honestly: run real verification only if the installed or injected `pyewf` API supports it safely; otherwise report `not_supported` or `not_run` with warnings.

Do not implement EWF-backed byte streams, partition parsing, filesystem parsing, file-content providers, file-list export, static HTML, search/timeline, UI, reports, deleted recovery, carving, packaging, or required native dependency installation.

## Required Behavior

Keep the public adapter shapes stable:

- `ReaderDependencyStatus`
- `EwfMetadataResult`
- `VerificationStatus`
- `EwfReaderWarning`
- `run_e01_intake()`
- `python -m app.backend.api.first_testing`

When `pyewf` is missing:

- behavior must stay dependency-safe;
- no crash;
- no required native dependency;
- status remains honest, such as `metadata_unavailable`;
- verification remains `not_run`;
- existing tests continue to pass.

When a fake or real `pyewf` module is available:

- open the discovered segment set read-only or through the safest read-only API available;
- normalize best-effort metadata fields when exposed;
- keep missing optional fields as warnings, not fatal errors;
- report `reader_open_failed` or equivalent structured warning on open failure;
- close handles deterministically;
- do not treat stored hash metadata as verification success;
- run a real verification/check only if the API supports a clear integrity check;
- otherwise return `not_supported` with `verification_not_supported`.

Suggested normalized metadata keys:

- `format`
- `segment_count`
- `media_size`
- `bytes_per_sector`
- `case_number`
- `description`
- `examiner`
- `evidence_number`
- `acquired_date`
- `system_date`
- `hashes`
- `reader`

Sensitive fields may remain in local JSON artifacts, but shared summaries/docs must mention redaction requirements.

## Tests

Add dependency-free tests. Do not require real E01 files, real `pyewf`, libewf, or native build tools.

Required coverage:

- missing `pyewf` behavior remains dependency-unavailable and existing first-testing smoke behavior still passes;
- injected fake `pyewf` reader returns normalized metadata;
- partial metadata produces metadata plus warnings;
- reader open failure produces structured `reader_open_failed` behavior;
- stored hash metadata does not become verification success;
- fake verification success maps to `verification_ok`;
- fake verification failure maps to `verification_failed`;
- fake verification exception maps to `verification_error`;
- unsupported verification API maps to `not_supported`;
- first-testing command writes metadata/verification artifacts without creating later-slice artifacts;
- `python -m app.backend.api.first_testing` with the local dependency-unavailable path still works against dummy `.E01` files.
