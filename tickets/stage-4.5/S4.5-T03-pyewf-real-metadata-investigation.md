# S4.5-T03 - Real Pyewf Metadata And Verification Plan

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Plan the smallest safe path to reading real EWF metadata and verification status from user-provided E01 files through `pyewf`/libewf.

This ticket is documentation-only. It does not install `pyewf`, modify the adapter, or implement real metadata/verification behavior.

## Current Local Dependency State

Observed on 2026-07-15:

```powershell
python -c "import importlib.util; spec=importlib.util.find_spec('pyewf'); print('available' if spec else 'missing')"
# missing
```

Current project behavior is therefore still the dependency-safe path:

- `PyewfEwfReaderAdapter` reports `dependency_unavailable` when `pyewf` cannot be imported.
- If a fake/importable `pyewf` module is injected, the adapter reports `real_reader_not_implemented`.
- `run_e01_intake()` maps those adapter states to `metadata_unavailable` or `reader_not_implemented`.
- Default tests must continue to pass without `pyewf`, libewf, real E01 evidence, or native build tools.

## Dependency Investigation Plan

Before implementation, confirm:

- whether `pyewf` can be installed cleanly on the user's Windows environment;
- whether it is available as a wheel or requires native build tooling;
- whether libewf DLLs need to be discoverable on `PATH`;
- how to report package version and libewf version, if exposed;
- whether imports work from the same Python environment used for `python -m pytest`;
- whether opening an E01 is read-only by default and whether any write-capable handles exist that must be avoided;
- whether multi-segment opens require all segment paths or only the selected `.E01` path.

Do not make `pyewf` required for default tests. If installation is not clean, Stage 4.5 should keep the adapter dependency-optional and continue with explicit unavailable statuses.

## Metadata Scope

The first real metadata implementation should attempt only fields that are clearly exposed and low-risk. Suggested normalized keys:

| Normalized key | Meaning | Notes |
| --- | --- | --- |
| `format` | Evidence format label | Keep as `EWF` or reader-provided equivalent |
| `segment_count` | Count of segment paths passed to adapter | Can come from discovery even if reader has no separate value |
| `media_size` | Total logical media/image size in bytes | Needed later for EWF-backed stream planning |
| `bytes_per_sector` | Sector size if exposed | Useful for later volume/filesystem work |
| `case_number` | Acquisition case number if present | Sensitive; redact in shared summaries |
| `description` | Evidence description if present | Sensitive; redact in shared summaries |
| `examiner` | Examiner/acquirer if present | Sensitive; redact in shared summaries |
| `evidence_number` | Evidence id/number if present | Sensitive; redact in shared summaries |
| `acquired_date` | Acquisition timestamp if present | Preserve original string plus parser status |
| `system_date` | System timestamp if present | Preserve original string plus parser status |
| `hashes` | EWF-stored hash values if exposed | Store as metadata only; do not treat as verified unless verification uses it |
| `reader` | Reader name/version | Helps explain parser behavior |

Metadata should be best-effort. Missing optional fields should not fail the entire intake if the image opens successfully.

## Verification Scope

Do not conflate metadata reading with verification.

Verification should answer a narrower question: did the adapter run an integrity check against the EWF evidence and what was the result?

Recommended verification statuses:

| Status | Supported | Meaning |
| --- | --- | --- |
| `not_run` | false | Verification was skipped because dependency/input/opening was unavailable |
| `not_supported` | false | Current adapter cannot perform verification with the installed dependency |
| `verification_ok` | true | Adapter ran verification and it passed |
| `verification_failed` | true | Adapter ran verification and reported failure |
| `verification_error` | true | Adapter attempted verification but errored before a trusted pass/fail result |
| `verification_partial` | true | Adapter verified only part of the segment set or returned incomplete details |

If only stored hash values are readable but no verification check was run, the result should remain `not_run` or `not_supported`; the hashes belong in metadata with a warning such as `stored_hash_not_verified`.

## Result Shape Mapping

Preserve the existing adapter contracts where possible:

- `ReaderDependencyStatus`
  - keep `name="pyewf"`;
  - set `available=True` only when import and basic handle construction/open support are usable;
  - populate `version` when exposed.
- `EwfMetadataResult`
  - keep `adapter_name="pyewf-reader"`;
  - keep `read_only=True`;
  - store normalized metadata in `metadata`;
  - include all selected segment paths in `source_paths`;
  - include warnings for partial/missing fields.
- `VerificationStatus`
  - use the statuses above;
  - put adapter-specific details in `details`, not in top-level fields.
- `EwfReaderWarning`
  - use stable warning codes so command summaries and review notes can group them.

## Proposed Warning Codes

Recommended codes:

| Code | Meaning |
| --- | --- |
| `dependency_unavailable` | `pyewf` cannot be imported |
| `dependency_version_unknown` | Import works but version is not exposed |
| `segment_missing` | Segment discovery found missing sibling segments before reader open |
| `reader_open_failed` | Adapter could not open the supplied segment set |
| `metadata_field_unavailable` | Optional metadata field was not exposed |
| `metadata_partial` | Some metadata was read, but one or more expected fields failed |
| `stored_hash_not_verified` | Hash values were read as metadata only |
| `verification_not_supported` | Installed reader has no usable verification path |
| `verification_failed` | Verification ran and failed |
| `verification_error` | Verification attempted but errored |
| `read_only_assumed` | Adapter API does not expose mode clearly, but project used the read-only path available |

Expected reader exceptions should become structured statuses and warnings, not raw tracebacks.

## Automated Test Strategy

Default tests should use fake injected `pyewf` modules/handles and tiny dummy segment filenames. They should not require real E01 files.

Coverage targets for a future implementation ticket:

- missing `pyewf` still returns the existing dependency-unavailable result;
- importable fake reader can return normalized metadata fields;
- metadata partial failure returns metadata plus `metadata_partial`;
- open failure returns empty metadata and `reader_open_failed`;
- stored hash values do not become verification success by themselves;
- verification success maps to `verification_ok`;
- verification failure maps to `verification_failed`;
- verification exception maps to `verification_error`;
- no default test writes to evidence paths;
- `run_e01_intake()` status mapping remains honest and does not report `ok` when real reader behavior failed or is unsupported.
