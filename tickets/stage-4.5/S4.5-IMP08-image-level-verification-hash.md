# S4.5-IMP08 - Image-Level Verification Hash Artifact

Status: Review

Stage: Stage 4.5 - first testing implementation runway extension

Owner: VS Code coding agent

Reviewer: Research/review agent

## Objective

Add an independent, first-testing image-level hash artifact for the full logical EWF image.

The user tested the Stage 4.5 demo and identified a real gap: the demo shows metadata and parser status, but it does not yet produce a verification-style hash over the entire image. This ticket must add that capability before Stage 5 search/timeline resumes.

This must be a computed hash over the EWF logical image stream, not:

- a hash of the `.E01/.E02/...` segment files as stored on disk;
- stored EWF hash metadata copied from the image;
- a claim that unsupported `pyewf` verification succeeded;
- a selected-file hash;
- a fixture/stub hash labeled as real evidence.

## Scope Boundary

Allowed:

- add a chunked image-hash service over the existing `ImageByteStream` / `EwfImageByteStream` interface;
- add first-testing CLI options to explicitly request full-image hashing;
- write `image-hash.json` and update manifest/summary/unsupported-section inventory;
- update tests and documentation for image-level hashing;
- run a real-image hash command only if it can complete honestly.

Not allowed:

- no Stage 5 search/timeline work;
- no UI/reporting system/PDF;
- no deleted recovery or carving;
- no broad filesystem crawl;
- no selected-file auto-selection;
- no real evidence commits;
- no pretending stored EWF metadata is an independent verification hash.

## Required Behavior

Add explicit CLI behavior rather than making every first-testing run hash a large image by default.

Suggested command shape:

```powershell
.\.python312-embed\python.exe -m app.backend.api.first_testing --evidence-dir ".\ Test Image" --first-segment "C16242-1-RL1-E003.E01" --case ".test-artifacts\first-testing\image-hash-real-image" --output ".test-artifacts\first-testing\image-hash-real-image\outputs" --hash-image --redact-paths
```

Minimum options:

- `--hash-image`: opt in to full logical-image hashing.
- `--image-hash-algorithm sha256`: optional; default may be `sha256`.

Optional if useful:

- `--image-hash-chunk-size`.
- progress fields written periodically to the output artifact.

The implementation must:

- open the evidence read-only through the existing EWF-backed logical image stream;
- read the full logical media size sequentially in bounded chunks;
- compute at least SHA-256;
- record byte count hashed;
- record logical media size;
- record whether byte count equals logical media size;
- record source path/provenance with redaction in shared summaries;
- preserve `source_modified: false` and `read_only_asserted: true`;
- keep missing dependency or unsupported stream states as structured statuses, not crashes.

## Artifact

Write:

- `outputs/image-hash.json`

Minimum fields:

- `status`: `not_run`, `hashing`, `completed`, `failed`, `dependency_unavailable`, `stream_unavailable`, or `interrupted`;
- `algorithm`;
- `hexdigest` when completed;
- `bytes_hashed`;
- `logical_media_size`;
- `byte_count_matches_media_size`;
- `source_kind`: for the real path, `ewf_logical_image`;
- `adapter_name`;
- `segment_count`;
- `started_at` and `completed_at` if available;
- `warnings`;
- `provenance`;
- `read_only_asserted`;
- `source_modified`;
- `privacy_note`.

The command summary should show only non-sensitive values:

- image hash status;
- algorithm;
- bytes hashed;
- whether byte count matched media size;
- hexdigest only if the user/reviewer considers it acceptable to share.

## Implementation Notes

- Added `--hash-image`, `--image-hash-algorithm`, and `--image-hash-chunk-size` to the first-testing command.
- Default runs write `outputs/image-hash.json` with status `not_run`.
- Requested runs compute SHA-256 over the read-only EWF logical image stream in chunks and record status, digest, byte counts, logical media size, byte-count match, provenance, warnings, read-only assertion, and source-modified assertion.
- Stored EWF hash metadata, segment container files, selected-file hashes, and stub bytes are not treated as the independent image-level hash.
- Missing dependency, stub-adapter, and stream failures return structured statuses rather than fake completion.
- Focused verification: `.\.python312-embed\python.exe -m pytest app\tests\test_image_stream.py app\tests\test_first_testing_command.py` reported 31 passed in 32.90s.
- Full verification: `.\.python312-embed\python.exe -m pytest` reported 189 passed in 87.61s.
- The full local real-image hash command was not run to completion in the coding-agent session because the existing reviewed real-image smoke reports a logical media size of 1,024,209,543,168 bytes. The local first segment exists; reviewer/user can run the command in the Real-Image Requirement section when ready. Stage 5 remains blocked until the reviewer accepts this outcome and S4.5-IMP09/S4.5-IMP10 are also reviewed.
