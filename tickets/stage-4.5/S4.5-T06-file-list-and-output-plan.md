# S4.5-T06 - File List And Output Plan

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Plan file-list export and the result surfaces for the first-testing command: command prompt summary, JSON artifacts, CSV file list, and optional static HTML.

This ticket is documentation-only. It does not implement output rendering, file-list export, or static HTML generation.

## Output Goal

Stage 4.5 should give the user something inspectable beyond automated tests:

- concise command prompt summary;
- local JSON artifact bundle;
- CSV file-list export for spreadsheet-style review;
- export manifests when selected-file export is supported;
- optional static HTML summary for visual review;
- explicit unsupported/dependency states when real parser pieces are not available.

These outputs are not Stage 5 search/timeline. They are first-testing artifacts that make the current progress visible and reviewable.

## JSON Artifact Plan

Recommended artifact ownership:

| Artifact | Source | Purpose |
| --- | --- | --- |
| `run-manifest.json` | First-testing command | Top-level run status, arguments, versions, artifact paths, warnings, unsupported sections |
| `command-summary.txt` | First-testing command | Human-readable summary matching command prompt output |
| `intake.json` | `run_e01_intake()` | Segment discovery, adapter status, metadata/verification shape |
| `case.json` | Case-store helpers | Case id, evidence id, database path, schema version |
| `audit.json` | `list_audit_events()` | Manual run audit trail |
| `segment-discovery.json` | `discover_e01_segments()` / intake result | Segment list and missing/gap warnings |
| `metadata.json` | `EwfMetadataResult` | Real or unavailable metadata state |
| `verification.json` | `VerificationStatus` | Real, not-run, unsupported, or failed verification state |
| `volumes.json` | `discover_volumes()` | Volume discovery result and parser status |
| `filesystems.json` | `FilesystemResult` / `list_directory()` | Filesystem parser/listing status |
| `file-list.json` | `FilesystemEntry.to_dict()` records | Full JSON file list with provenance |
| `selected-file-preview.json` | `preview_file()` | Raw/text/hex preview result when supported |
| `selected-file-analysis.json` | Stage 4 analysis functions | Hash/signature/mismatch/known-file results when supported |
| `selected-file-export.json` | `export_file()` | Export result and manifest references when supported |
| `unsupported-sections.json` | First-testing command | Explicit unsupported/deferred pieces |

All JSON should be stable, sorted where practical, and include schema/status/warning fields. Private local paths may remain in ignored local artifacts, but shared summaries should redact them.

## File-List JSON Fields

`file-list.json` should start from parser-backed `FilesystemEntry.to_dict()` records and include command/run context.

Suggested shape:

```json
{
  "schema_version": "stage4_5.file_list.v1",
  "status": {"code": "ok", "ok": true, "message": "File list exported."},
  "case_id": "case_...",
  "evidence_id": "evidence_...",
  "source_path": "<local path>",
  "volume_id": "volume-0",
  "filesystem_type": "ntfs",
  "adapter": {"name": "pytsk3-filesystem-adapter", "available": true},
  "entry_count": 0,
  "entries": [],
  "warnings": []
}
```

Each entry should preserve:

- case id when supplied;
- evidence id when supplied;
- source path;
- volume id;
- volume offset and length;
- filesystem type;
- adapter name;
- file id;
- path;
- name;
- entry type;
- size;
- allocated;
- deleted;
- timestamps;
- read-only assertion;
- status;
- warnings.

Do not use CSV file-list export as search/indexing. Stage 5 search/timeline may consume reviewed records later, but T06 output is a first-testing artifact only.

## Optional Static HTML Summary

Static HTML is allowed as a simple report artifact, not as UI work.

Recommended constraints:

- generate a single local `outputs/reports/summary.html`;
- no JavaScript required for the first version;
- no external network assets;
- no bundled evidence content;
- redacted shared version if paths or metadata are sensitive;
- clear status badges/labels for ok, partial, unavailable, unsupported, failed, and skipped;
- link or list local artifact filenames, not full private paths in shared screenshots.

Suggested sections:

- case/evidence summary;
- segment chain;
- dependency/capability summary;
- metadata and verification status;
- volume/filesystem status;
- file-list counts by entry type and parser status;
- selected-file preview/export/analysis status;
- warnings and unsupported sections;
- artifact inventory.

Do not turn static HTML into a full UI, interactive browser, search surface, or timeline.

## Redaction Rules

Shared command summaries, screenshots, and HTML previews should redact:

- evidence root as `<EVIDENCE_ROOT>`;
- user profile paths;
- case names unless approved;
- examiner/acquisition names;
- serial numbers;
- sensitive file paths or file names;
- acquisition notes and descriptions.

Local unredacted artifacts may stay under ignored `.test-artifacts/first-testing/` for repeatability.

## Automated Test Strategy

Default tests should remain dependency-free and real-evidence-free.

Future implementation tests should cover:

- JSON artifact writer refuses evidence/output overlap;
- file-list JSON preserves all required entry provenance fields;
- CSV writer emits stable header order and escapes commas/quotes/newlines;
- unsupported parser states still produce manifest/summary output;
- summary text includes dependency and unsupported states;
- redaction mode replaces evidence roots in summary/HTML;
- optional HTML escapes file names and metadata text;
- no output writer creates files beside evidence paths;
- no default test requires real E01 files or native dependencies.

## Acceptance Criteria

- The plan gives the user inspectable output beyond automated pytest.
- File-list export is clearly separate from search/indexing.
- Output formats preserve unsupported/not-implemented states.
- The implementation runway lines up S4.5 planning tickets into buildable follow-on tickets.
- Redaction and evidence-output separation are explicit.
