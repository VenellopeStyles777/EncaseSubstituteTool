# S4.5-T05 - E01-Backed File Content Provider Plan

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Plan how real filesystem entries from an E01 image will supply bytes to preview, export, hash, and signature analysis.

This ticket is documentation-only. It does not implement byte extraction or change provider APIs.

## Current Truth

The current project has three byte-consuming surfaces:

- `preview_file()` consumes `PreviewContentProvider` bytes and renders raw/text/hex previews.
- `export_file()` consumes `ExportContentProvider` bytes, writes output files, writes manifests, verifies SHA-256/byte count from the written artifact, and optionally records `ExportAuditContext` audit rows.
- `hash_file_content()` and `detect_file_signature()` consume `AnalysisContentProvider` bytes for hashes and signatures.

Current providers are stub/synthetic or explicit caller-provided bytes. No provider extracts real file content from E01-backed filesystem entries yet.

S4.5-T04 owns the prerequisite path:

```text
selected .E01 -> EWF-backed image stream -> VolumeInfo -> FilesystemEntry
```

S4.5-T05 starts only after that path can produce real parser-backed file entries.

## Provider Design Direction

Use a shared lower-level selected-file content reader plus thin provider wrappers.

Recommended shape:

```text
FilesystemEntry from real parser
  -> E01FileContentReader / SelectedFileContentReader
      -> Preview provider wrapper
      -> Export provider wrapper
      -> Analysis provider wrapper
```

Reason:

- Preview, export, and analysis have different result contracts and warning classes.
- A shared lower-level reader keeps the actual parser-backed byte extraction logic in one place.
- Thin wrappers can translate the same content-read outcome into `PreviewContent`, `ExportContent`, or `AnalysisContent`.
- This prevents preview-rendered text/hex or exported artifacts from becoming accidental analysis source bytes.

The shared reader should own:

- lookup of a parser-backed file handle from `file_id` and volume context;
- read-only byte extraction from that handle;
- source size and byte-count checks;
- sparse/partial/deleted/unreadable status decisions;
- parser/source warning details;
- limits for first-testing safety.

The wrappers should own only protocol-specific conversion.

## Current Function Reuse Plan

| Feature | Reuse | Provider wrapper |
| --- | --- | --- |
| Raw/text/hex preview | `preview_file()` | `E01PreviewContentProvider` returns bounded bytes as `PreviewContent` |
| Export selected file | `export_file()` | `E01ExportContentProvider` returns raw bytes as `ExportContent` |
| File hash | `hash_file_content()` | `E01AnalysisContentProvider` returns raw bytes as `AnalysisContent` |
| File signature | `detect_file_signature()` | Same `E01AnalysisContentProvider`; signature caller already bounds inspected bytes |
| Extension mismatch | `evaluate_extension_mismatch()` | No byte provider; consumes reviewed signature result plus metadata |
| Known-file matching | `match_known_file_hashes()` | No byte provider; consumes reviewed hash result |

Do not add alternate preview, export, hash, or signature implementations for E01 files. The first-testing command should route real content through the existing surfaces.

## Provider Identity

Recommended identity values:

| Surface | Provider name | Source kind | Synthetic/generated |
| --- | --- | --- | --- |
| Preview | `e01-preview-content-provider` | `real_parser` | false |
| Export | `e01-export-content-provider` | `real_parser` | false |
| Analysis | `e01-analysis-content-provider` | `real_parser` | false |

Provider output should carry parser details when available:

- parser name, such as `pytsk3`;
- parser version, if exposed;
- source name/version for the EWF reader if available;
- source status, such as `ok`, `partial`, or `content_source_unavailable`;
- source content size when known.

If a provider cannot prove that bytes came from the E01-backed real parser path, it must not label `source_kind` as `real_parser`.

## Provenance Requirements

Every byte-consuming operation must preserve:

- case id when supplied;
- evidence id when supplied;
- selected E01/source path;
- volume id;
- volume offset and length;
- filesystem type;
- filesystem adapter name;
- parser/source status;
- file id;
- file path;
- file name;
- entry type;
- logical file size;
- allocation/deleted state;
- timestamps;
- read-only assertion;
- provider name;
- provider source kind;
- parser/source warnings.

The first-testing command should show whether each output used `real_parser`, `stub`, `synthetic`, `generated_fixture`, `metadata_only`, or `content_source_unavailable`.

## Read Size And Streaming Risk

The current provider protocols return `bytes` in memory. That is acceptable for small stubs and bounded preview, but risky for large real evidence files.

Future implementation should choose one of these policies explicitly:

1. First-testing bounded content policy:
   - preview reads bounded ranges only;
   - signature reads bounded prefixes only;
   - hash/export refuse files above a documented size limit until streaming providers exist;
   - refused operations return structured `file_too_large_for_in_memory_provider`.

2. Streaming provider extension:
   - add streaming/chunked read support before real large-file export/hash;
   - keep existing APIs stable where possible;
   - update export/hash implementation to stream from parser bytes without loading the whole file.

Do not silently load large real E01-extracted files into memory in the first testing command.

## Failure Statuses And Warnings

Recommended shared content statuses:

| Status | Meaning |
| --- | --- |
| `ok` | Complete requested content was read |
| `content_source_unavailable` | No parser-backed content source exists for this entry |
| `path_not_file` | Entry is not a regular file |
| `metadata_only_source` | Entry metadata exists but no byte source is attached |
| `file_too_large_for_in_memory_provider` | Provider refuses full read under bounded first-testing policy |
| `file_content_partial` | Some bytes were returned, but content is incomplete |
| `file_content_unreadable` | Parser could not read the file content |
| `file_content_provider_error` | Provider raised an expected/handled error |
| `content_size_mismatch` | Bytes read do not match expected logical size |
| `sparse_file_unsupported` | Sparse layout cannot be represented safely yet |
| `sparse_file_partial` | Sparse file was read with holes or missing ranges |
| `deleted_entry_metadata_only` | Entry is deleted and only metadata is available |
| `deleted_recovery_unsupported` | Deleted recovery is out of scope with current adapter |
| `encrypted_or_compressed_unsupported` | Parser cannot read encrypted/compressed content safely |

Recommended warnings:

- `real_parser_content`;
- `parser_content_partial`;
- `parser_content_unreadable`;
- `content_size_mismatch`;
- `sparse_file_warning`;
- `deleted_recovery_deferred`;
- `large_file_refused`;
- `content_provider_exception`;
- `read_only_content_source`;
- `parser_uncertain_allocation_state`.

Provider exceptions should become structured non-ok results and warnings, not raw tracebacks in command output.

## Deleted, Sparse, And Special Files

Keep deleted recovery and carving out of scope unless a reviewed adapter exposes recoverable byte ranges.

Rules:

- allocated regular files may be read when the parser exposes complete content;
- directories and special files should return `path_not_file`;
- deleted entries with no recoverable content should return `deleted_entry_metadata_only` or `deleted_recovery_unsupported`;
- sparse files should either be represented with an explicit policy or refused as unsupported;
- symbolic links, device nodes, alternate data streams, compressed files, and encrypted files should be explicit non-ok or deferred statuses until intentionally supported.

Do not fake recoverable deleted bytes to make export or analysis appear complete.

## First-Testing Command Behavior

Once implemented, the first-testing command should distinguish:

- file metadata available but content unavailable;
- preview available but export/hash refused because of size;
- signature available from a bounded prefix while full hash/export is not run;
- export succeeded and was verified from written output;
- hash/signature used parser-backed source bytes, not stub bytes;
- extension mismatch consumed reviewed signature output, not a second read.

The command should never fall back to `StubPreviewProvider`, `StubExportContentProvider`, or `StubAnalysisContentProvider` while claiming E01-backed output. If real content is unavailable, report that visibly.

## Automated Test Strategy

Default tests should remain dependency-free and real-evidence-free.

Future implementation tests should use fake content readers and parser-backed-looking entries:

- preview wrapper returns bounded real-parser-labeled content;
- export wrapper returns real-parser-labeled content and existing `export_file()` writes/verifies it;
- analysis wrapper returns real-parser-labeled content and existing hash/signature functions consume it;
- missing content maps to surface-specific non-ok statuses;
- directory entries return `path_not_file`;
- deleted metadata-only entries return deleted recovery deferred status;
- large files are refused if bounded policy is selected;
- partial content emits warnings and non-ok or partial status;
- provider exceptions become structured provider-error statuses;
- no E01 content path falls back to stub providers in real-parser tests;
- provenance fields survive wrapper conversion into preview/export/analysis results.

Optional real E01 manual tests must be local-only and skipped unless the user supplies evidence paths explicitly.

## Future Implementation Sequence

Recommended later implementation order:

1. Add a shared selected-file content reader contract with fake tests.
2. Add `E01PreviewContentProvider` wrapper for bounded preview.
3. Add `E01AnalysisContentProvider` wrapper for bounded signature and small-file hash.
4. Add `E01ExportContentProvider` wrapper only after size policy is explicit.
5. Wire the first-testing command to run one selected-file preview/export/hash/signature path.
6. Add optional manual E01 transcript once the user provides files.

If streaming is required before full export/hash, split that into a separate ticket instead of stretching T05.

## Acceptance Criteria

- The plan shows exactly how current preview/export/analysis functions will be reused.
- The plan prevents preview/export/analysis from using stub bytes when the command claims E01-backed output.
- The plan preserves provenance and warnings for every byte-consuming operation.
- The plan defines memory/streaming risk before real large-file reads.
- The plan keeps deleted recovery, carving, search/timeline, reporting, packaging, and UI out of scope.
