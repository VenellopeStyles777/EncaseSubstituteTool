# S4.5-T07 - Workflow, Guardrail, And Review Optimization

Status: Review

Stage: Stage 4.5 - First testing

## Objective

Optimize the current agent workflow for manual testing with real user-provided E01 files.

This ticket is documentation-only. It defines the review gates for the future Stage 4.5 implementation runway.

## Why This Exists

Stage 4.5 is moving from dependency-free backend foundations toward commands the user can run against real local E01 files. That changes the risk profile:

- user-provided evidence may contain sensitive paths and metadata;
- native parser dependencies may be missing, partial, or unsafe to require by default;
- command output may tempt overclaiming real parser support;
- manual test artifacts can accidentally leak case details;
- future implementation tickets need a repeatable review checklist, not just ad hoc judgment.

T07 defines the guardrails that future implementation tickets should follow before Stage 5 search/timeline work begins.

## Stage 4.5 Implementation Review Gates

Each future implementation slice should pass these gates:

| Gate | Applies to | Required proof |
| --- | --- | --- |
| Scope gate | Every implementation ticket | Ticket id, allowed files, explicit out-of-scope list, no Stage 5 work |
| Evidence safety gate | Any real/local evidence path handling | Read-only access, no writes beside evidence, output/evidence overlap refusal |
| Dependency gate | `pyewf`, libewf, `pytsk3`, TSK work | Missing dependency stays structured and default tests still pass |
| Provenance gate | Parser/listing/content/output work | Source path, evidence id when supplied, volume/file ids, parser/source status, warnings |
| Output honesty gate | Command, JSON, CSV, HTML, docs | Unsupported/partial/unavailable states remain visible |
| Test gate | Every behavior change | Automated tests plus optional manual result or documented skip |
| Privacy gate | Manual transcripts/screenshots/HTML | Private paths and sensitive metadata redacted before sharing |
| Manual-tested gate | `functionality.md` manual status | User-confirmed real E01 run required before marking `Tested` |

## Future Slice Checklist

### S4.5-IMP01 - Command Shell And Case Workspace

Required checks:

- command accepts direct `.E01` and directory-plus-first-segment forms;
- `.E02+` primary input is rejected with guidance;
- case database/output workspace is created outside evidence paths;
- intake result persists to SQLite only through explicit case-store helpers;
- command writes run manifest and unsupported-section output;
- no real parser support is claimed.

Manual test note:

- real E01 manual run may prove segment discovery and workspace creation only;
- metadata, verification, volume, filesystem, preview, export, hash/signature should remain unsupported until later slices land.

### S4.5-IMP02 - Real Metadata And Verification

Required checks:

- `pyewf` missing remains dependency-unavailable, not a crash;
- metadata reading and verification are separate statuses;
- stored hash values are not treated as verification success;
- sensitive metadata fields are redacted in shared summaries;
- default tests use fakes/mocks only.

Manual test note:

- user must approve any real E01 metadata/verification run;
- shared notes must not expose case number, examiner, evidence number, acquisition notes, or private paths unless explicitly approved.

### S4.5-IMP03 - EWF Stream, Partition, Filesystem

Required checks:

- `.E01` is not treated as a raw local file;
- EWF stream implements existing stream shape and is read-only;
- partition offsets/lengths are byte values;
- filesystem entries preserve parser status and read-only assertions;
- dependency-unavailable and real-parser-not-implemented states remain visible.

Manual test note:

- file navigation is manually tested only when real parser-backed entries are displayed from a user-provided image.

### S4.5-IMP04 - File Content Providers

Required checks:

- no fallback to stub providers when claiming E01-backed content;
- preview/export/hash/signature reuse existing functions;
- source kind is `real_parser` only when backed by parser bytes;
- large-file memory/streaming policy is explicit;
- deleted, sparse, special, encrypted, compressed, and partial-content states are structured.

Manual test note:

- selected-file preview/export/hash/signature are manually tested only when bytes come from a parser-backed file entry and the source kind/warnings prove it.

### S4.5-IMP05 - Output Bundle

Required checks:

- file-list JSON preserves complete provenance;
- CSV columns are stable and escaped;
- summary text names unsupported and dependency states;
- optional static HTML escapes user-controlled text and does not fetch network assets;
- local artifacts stay outside evidence paths.

Manual test note:

- screenshots or HTML previews must be redacted before sharing.

### S4.5-IMP06 - Final Guardrail And Handoff

Required checks:

- docs agree with implemented behavior;
- `functionality.md` manual-test status reflects only user-confirmed runs;
- review notes distinguish automated, mocked, manual, skipped, and unavailable testing;
- Stage 5 remains deferred unless the user explicitly moves on.

## Required Implementation Handoff Format

Every Stage 4.5 coding-agent handoff should include:

```text
Ticket:
Summary:
Files changed:
Behavior changed:
Artifacts created:
Dependencies installed: yes/no
Real E01 used: yes/no
Evidence path redacted as:
Source modified: no
Output path:
Automated tests:
Mocked dependency tests:
Manual E01 test:
Skipped manual pieces:
Unsupported/deferred states shown:
Privacy/redaction notes:
Known risks:
Next recommended ticket:
```

If no real E01 was used, say so directly. Do not describe automated tests or fake parser tests as manual E01 testing.

## Review Note Categories

Research/review notes should distinguish:

- `Automated tests`: normal dependency-free test commands such as `python -m pytest`.
- `Mocked dependency tests`: fake `pyewf`/`pytsk3` or fake parser objects.
- `Manual E01 tests`: user-approved command runs against real local E01 files.
- `Skipped manual tests`: manual steps not run, with reason.
- `Unavailable tests`: blocked by missing dependency, missing evidence, or missing implementation.
- `Privacy notes`: redactions applied or sensitive output withheld.

## Functionality Manual-Test Status Rule

`functionality.md` manual-test status should remain `Untested` unless all are true:

- the relevant behavior is implemented, not just planned;
- the user provides or approves a real local E01/manual test input where required;
- the command is run and produces inspectable output;
- the source evidence is confirmed unmodified;
- output artifacts are generated under an allowed output path;
- shared results are redacted or approved for disclosure;
- the user confirms the manual result is acceptable.

Mocked tests, stub providers, generated dummy filenames, and dependency-unavailable output do not count as manual testing with actual E01 evidence.

## Privacy And Redaction Rules

Before pasting transcripts, screenshots, HTML summaries, or file-list excerpts:

- replace evidence roots with `<EVIDENCE_ROOT>`;
- remove or replace user profile names;
- redact case names, client names, examiner names, evidence numbers, serial numbers, and acquisition notes unless approved;
- redact sensitive file names and paths when content is private;
- avoid sharing full file lists from real evidence unless the user explicitly approves;
- keep unredacted local artifacts under ignored output locations.

## No-Overclaim Review Checklist

Reviewers should reject or request changes when output or docs imply:

- real EWF metadata was read when the adapter only reported dependency status;
- stored hash metadata equals verified image integrity;
- `.E01` segment bytes were treated as raw image bytes;
- partition/filesystem output is real when it came from a stub;
- preview/export/hash/signature used stub bytes while command text says E01-backed;
- a partial, sparse, deleted, encrypted, unsupported, or failed read is reported as complete success;
- CSV/HTML/file-list output is treated as search/indexing;
- Stage 5 search/timeline begins before Stage 4.5 implementation is reviewed or explicitly set aside.

## Evidence Safety Checklist

Future implementation and review should confirm:

- evidence inputs are opened read-only;
- no command writes beside `.E01/.E02/...` segments;
- output directory is not inside evidence directory;
- evidence directory is not inside output directory;
- local config and artifacts stay under ignored paths unless user chooses another safe output;
- delete/move/repair/resequence operations are not performed on evidence;
- source modification is reported as `no` in handoff.

## Acceptance Criteria

- Future Stage 4.5 tickets have stronger review guardrails than earlier backend-only stages.
- Manual E01 testing can be recorded without committing evidence or leaking sensitive data.
- Review agents have a repeatable checklist for command output, docs, and forensic-soundness claims.
- The implementation runway from S4.5-T01 through S4.5-T08 has explicit review gates.
- `functionality.md` manual-test status rules are clear.
